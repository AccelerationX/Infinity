# 01 LLM Agent 能力评测框架 —— 面试完全指南

---

## 一、项目一句话定位

这是一个**面向 LLM Agent 的多维度自动化评测系统**，能够对 Agent 在推理、工具调用、代码生成、长上下文等维度的能力进行定量评分，并对失败案例做根因归因。

---

## 二、核心技术栈

- **Python 3.10** + **Pydantic**（结构化数据校验）
- **OpenAI API / 本地 vLLM**（被测模型接入）
- **Jinja2**（Prompt 模板管理）
- **Pytest**（自动化测试与断言）
- **Pandas + Matplotlib**（评分统计与可视化）

---

## 三、核心原理

传统的大模型评测（如 MMLU、GSM8K）只测"文本输出"，但 Agent 的核心能力在于**多轮交互 + 工具调用 + 任务完成度**。本项目设计的评测框架包含三层：

1. **任务层（Task）**：定义标准化的 Agent 任务，如"查天气并总结"、"写 Python 脚本并运行"。
2. **指标层（Metric）**：对每个任务设计可自动校验的通过标准，包括：
   - **Exact Match**：输出是否包含正确答案
   - **Tool Call Accuracy**：工具调用次数、参数是否正确
   - **Task Completion**：最终状态是否达成目标
   - **Latency & Token Usage**：效率指标
3. **归因层（Attribution）**：对失败案例自动分类，如"Prompt 理解错误"、"工具参数格式错误"、"推理链断裂"等。

## 四、实现细节

### 架构设计
```
tasks/           # 任务定义（JSON/YAML）
benchmark/       # 评测执行引擎
models/          # 模型适配器（OpenAI、Anthropic、本地模型）
reports/         # 评分报告生成
```

### 关键模块
- `BenchmarkRunner`：批量加载任务，控制并发调用被测 Agent
- `ModelAdapter`：统一封装不同 LLM 的 API 差异
- `MetricEvaluator`：针对每个任务类型注册对应的评分函数
- `FailureAnalyzer`：基于规则 + 少量 LLM 调用，对失败输出做根因分类

### 难点与解决
- **不同模型输出格式不一致**：要求 Agent 输出遵循结构化 JSON Schema，但模型经常"废话连篇"。解决方式是加了**输出后处理层**（用正则提取 JSON 块 + Pydantic 校验失败时 fallback 到 LLM 重格式化）。
- **工具调用评测需要模拟环境**：设计了 `MockToolExecutor`，在不调用真实 API 的情况下验证 Agent 是否生成了正确的工具调用序列。

## 六、简历描述建议

> **独立设计并实现了一套 LLM Agent 多维度能力评测框架**，支持对推理、工具调用、代码生成等任务进行自动化评分与失败归因。框架采用模块化架构，通过 `ModelAdapter` 统一接入 OpenAI、Anthropic 及本地 vLLM，利用 `MockToolExecutor` 在无真实 API 环境下验证工具调用正确性，最终输出多模型对比雷达图与失败模式分布报告。使用该框架完成了对 3 个主流模型的横向评测，识别出工具参数格式错误是最主要的失效模式（占比 35%）。

## 七、高频面试问题与回答

### Q1：你们的评测指标和传统的大模型评测（如 C-Eval）有什么区别？
**A**：C-Eval 测的是单轮知识问答，而我们的框架测的是**Agent 在交互过程中的综合能力**。我们不仅看最终答案对不对，还要看：
1. 中间有没有调用正确的工具；
2. 工具参数填得对不对；
3. 多轮对话里有没有丢失上下文；
4. 完成同样任务的 Token 消耗和延迟。

### Q2：如果模型输出不是标准的 JSON，你们怎么评分？
**A**：我们设计了一个 `OutputSanitizer` 层。第一步用正则尝试提取 JSON 块；第二步如果失败，会用一个小模型（或规则）做格式化重试；第三步如果仍然无法解析，就记录为"格式错误"并计入失败归因。

### Q3：怎么保证评测任务本身不带偏见？
**A**：任务设计遵循三个原则：
1. **可自动验证**：每个任务都有明确的通过标准，不依赖人工主观判断；
2. **多样性**：覆盖推理、代码、工具调用、长文本、安全性等多个维度；
3. **避免数据泄漏**：任务中的数据是项目期间手工合成或基于公开数据集二次扰动生成的。

### Q4：这个项目最大的挑战是什么？
**A**：最大的挑战是**不同模型的 Agent 输出行为差异极大**。有的模型喜欢先解释再输出 JSON，有的模型会漏掉必要的工具调用，还有的在多轮对话中会重复调用同一工具。我们通过大量的 Prompt Engineering 和后处理规则，最终把评测的可用率（能成功解析并评分的比例）从 60% 提升到了 95% 以上。
