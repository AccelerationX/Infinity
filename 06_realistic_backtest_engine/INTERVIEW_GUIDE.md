# 06 防前视偏差回测引擎 —— 面试完全指南

## 一、项目一句话定位

这是一个**面向量化策略的防前视偏差（Look-ahead Bias Free）回测引擎**，通过 Walkforward 验证、执行仿真和回测-实盘归因，确保策略收益评估结果的真实性和可复现性。

## 三、核心原理

传统回测引擎最容易犯的错误是**前视偏差**（Look-ahead Bias）——策略在做第 T 天的决策时，不小心使用了第 T+1 天才能知道的信息（比如收盘价、财报数据）。这会导致回测收益被严重高估。本引擎的核心设计原则：

1. **严格的数据时点管理**：
   每个数据点都必须标注其**可用时间戳（available_at）**。例如，日 K 线的收盘价要到当天收盘后才可用，所以 `available_at = 当日 15:00:00`。

2. **Walkforward 验证**：
   将历史数据划分为多个不重叠的窗口。在每个窗口内，只允许使用窗口起始点之前的数据训练/拟合参数，窗口内的数据仅用于测试。这样可以真实模拟策略在不同历史阶段的泛化能力。

3. **执行仿真（Execution Simulation）**：
   不是简单地用收盘价成交，而是考虑：
   - **滑点（Slippage）**：根据成交量和盘口深度估算
   - **市场冲击（Market Impact）**：大单对价格的临时影响
   - **延迟（Latency）**：从信号生成到订单到达交易所的时间延迟
   - **无法成交（No-fill）**：涨跌停、流动性不足时的部分或全部失败

4. **回测-实盘归因（Backtest vs Live Attribution）**：
   记录每一笔交易的预期价格 vs 实际成交价，拆解出**信号衰减、执行滑点、机会成本**等差异来源。

## 四、实现细节

### 架构设计
```
engine/
  ├── data_manager.py        # 数据加载与可用时点管理
  ├── clock.py               # 事件驱动时钟
  ├── strategy.py            # 策略基类
  ├── broker.py              # 撮合引擎（含滑点、延迟、冲击模型）
  ├── portfolio.py           # 账户与持仓管理
  ├── walkforward.py         # Walkforward 框架
  └── attribution.py         # 回测-实盘归因分析
```

### 关键模块
- `DataManager`：强制要求每列数据都有 `available_at` 字段。策略在 `get_data(until=t)` 时，只能拿到 `available_at <= t` 的数据。
- `Broker`：实现了三种撮合模式——
  - **Ideal**：完全按目标价成交（仅用于对比）；
  - **VWAP**：按当日成交量加权均价成交；
  - **Realistic**：结合订单簿深度模型、滑点函数和延迟分布进行概率化撮合。
- `WalkforwardEngine`：支持 expanding window 和 rolling window 两种模式，每个 window 独立运行策略，最终拼接出完整的 OOS（Out-of-Sample）收益曲线。

### 难点与解决
- **数据对齐复杂**：A 股不同股票的停牌日不同，导致截面数据常有缺失。我们采用**前向填充（forward fill）+ NaN 标记**策略，确保策略在缺失数据时收到明确的 `unavailable` 信号而不是隐式的旧数据。
- **Tushare 数据频率限制**：免费版 Tushare 只有日线数据。我们在引擎中预留了分钟线接口，验证时用日线回测，实盘对接时平滑切换到分钟线。

## 七、高频面试问题与回答

### Q1：什么是前视偏差？你们的引擎是怎么防止的？
**A**：前视偏差是指策略在做第 T 天决策时，使用了 T+1 及以后才能知道的信息。最常见的例子是用**当日收盘价**作为决策依据并在当日成交，但实际上收盘前是不知道收盘价的。

我们的引擎通过两个机制防止：
1. **数据时点标注**：每条数据都有 `available_at` 字段，策略只能查询 `available_at <= 当前时间` 的数据；
2. **延迟撮合**：即使策略在 T 日 14:55 生成了信号，Broker 也会根据设置的延迟参数，最早在 T+1 日开盘时才撮合成交。

### Q2：Walkforward 和简单的 Train/Test Split 有什么区别？
**A**：简单 Split 只分一次，无法模拟策略在**不同市场环境**下的表现。Walkforward 是把历史切成多个窗口，每个窗口都做一次"用过去训练、用未来测试"，最后把所有测试段拼起来。这样可以观察到策略参数是否稳定、是否存在过拟合、以及在牛市/熊市/震荡市中的表现差异。

### Q3：你们的撮合引擎和现实交易还差多少？
**A**：我们的 Realistic 撮合模型考虑了滑点、延迟和市场冲击，但仍然和真实交易有差距，因为我们没有真实的 Level-2 订单簿数据。差距主要来自：
1. **订单簿深度未知**：我们只能根据日均成交量做经验估计；
2. **没有考虑其他竞争策略的影响**：实盘里可能有很多机器人在同一时刻抢单；
3. **没有考虑停牌、复牌、除权除息的微观影响。

我们正在把这些因素逐步加入引擎，后续会接入真实的 Level-2 数据做校准。

### Q4：如果策略是高频的（比如分钟级调仓），你们的引擎还适用吗？
**A**：适用，但数据精度需要提升。我们引擎的架构是按事件时间轴驱动的，频率本身不是瓶颈。分钟级策略的问题在于需要分钟线数据来准确估计滑点。如果用日线数据来估计分钟级策略的滑点，误差会非常大。我们在引擎中预留了分钟线数据接口，数据接入后可以直接升级。
