# MACP v2 - 面试Q&A

> 常见问题及回答策略

---

## 1. 项目概述类

### Q1: 请介绍一下这个项目

**回答要点：**
- MACP = Multi-Agent Collaboration Protocol
- 解决单Agent能力边界问题，实现多Agent协作
- 核心特点：通用框架（不限于软件）+ DAG调度 + Web监控

**建议回答：**
"这个项目是一个通用多Agent协作框架。核心思路是把复杂任务拆解给不同角色的Agent并行处理，比如软件研发场景下，PM分析需求、架构师设计系统、前后端开发并行编码、QA写测试用例，最后集成。

框架的核心设计是"领域模板"系统，通过模板定义角色和流程，引擎负责调度执行。这样新增领域只需要写模板，不需要改核心代码。

另外我还做了Web监控界面，可以实时看到每个Agent在做什么、整体进度多少、有没有报错需要人工介入。"

---

### Q2: 为什么做多Agent而不是强化单Agent？

**回答要点：**
- 单Agent上下文窗口有限
- 角色专业化（让前端专家做前端）
- 并行提效

**建议回答：**
"主要是三个原因：

第一，**上下文限制**。即使是128K上下文的模型，处理大型项目也不够，分而治之更合理。

第二，**专业化分工**。让专门做架构的Agent做架构、专门写前端的Agent写前端，比让一个通用Agent什么都做效果更好。

第三，**并行执行**。需求分析完成后，前端、后端、测试可以同时进行，大幅缩短总时间。

其实这也反映了人类软件团队的组织方式——没有人是全栈超人，都是团队协作。"

---

### Q3: 这个项目和AutoGen/MetaGPT有什么区别？

**回答要点：**
- AutoGen太灵活，缺乏结构化流程
- MetaGPT太特定（仅软件）
- MACP取中间：结构化 + 通用

**建议回答：**
"我调研过这几个框架：

**AutoGen** 是微软的，特点是灵活，Agent之间可以自由对话。但问题是太灵活了，对于需要严格流程控制的生产场景不够结构化。

**MetaGPT** 专注于软件开发，把SOP（标准操作流程）固化在代码里。问题是领域耦合太重，想用来做金融分析就要改很多代码。

**MACP** 的设计是**模板化的领域系统 + 通用的调度引擎**。软件研发是一个模板，金融分析可以写另一个模板，核心调度引擎完全不用改。这样既保证了流程的结构化，又保持了跨领域的通用性。

另外MetaGPT没有Web监控界面，我需要实时看到Agent状态，所以自己做了仪表盘。"

## 2. 架构设计类

### Q4: 讲讲整体架构？

**画图辅助回答：**

```
用户请求 → Framework
              ↓
    ┌─────────┼─────────┐
    ↓         ↓         ↓
 Template  Scheduler  AgentRegistry
    ↓         ↓         ↓
 角色定义   DAG执行   LLM Agent池
```

**关键点：**
1. **三层分离**：Template（做什么）、Engine（怎么做）、Agent（谁来做）
2. **DAG调度**：基于依赖关系的任务编排
3. **状态持久化**：Job状态保存到本地，可恢复

---

### Q5: 为什么要分离Template和Engine？

**回答要点：**
- 单一职责
- 扩展性
- 团队协作

**建议回答：**
"这是这个项目最核心的设计决策。

**职责分离**：Engine只关心'如何调度'——DAG执行、Agent分配、错误处理；Template只关心'做什么'——有哪些角色、任务怎么分解。两者通过接口交互。

**扩展性**：新增一个领域（比如金融分析），只需要实现Template接口（大概200行代码），Engine完全不用动。

**团队协作**：业务专家可以独立设计Template，不用理解Engine代码；工程师优化Engine时也不用关心业务逻辑。

没有这层抽象的话，每新增一个领域都要改核心代码，很快项目就会变成'大泥球'。"

---

### Q6: DAG调度是怎么实现的？

**回答要点：**
- 状态机驱动
- 动态找就绪任务
- 线程池执行

**建议回答：**
"DAG调度的核心算法是这样的：

1. **状态机**：每个任务有状态——PENDING、RUNNING、COMPLETED、FAILED

2. **找就绪任务**：每次循环找出所有依赖已完成的任务
   ```python
   ready = [t for t in tasks if all(d in completed for d in t.dependencies)]
   ```

3. **并发控制**：用ThreadPoolExecutor限制并行度，避免同时调太多API

4. **执行监控**：任务完成后更新状态，保存到Workspace，触发事件

这个设计的好处是**天然支持并行**——只要依赖满足，多个任务可以同时执行。比如前端开发、后端开发、测试用例编写这三件事可以在架构设计完成后同时进行。"

---

### Q7: 如果某个Agent失败了怎么办？

**回答要点：**
- 错误隔离
- 状态记录
- 人工介入
- 重试机制

**建议回答：**
"我做了多层容错：

**错误隔离**：单个任务失败不会导致整个Job崩溃，只是标记该任务FAILED，其他任务继续执行。

**状态记录**：失败原因、错误信息都会记录到Job对象里，持久化到文件。

**告警通知**：通过EventBus发送事件，Web界面会显示红色告警，提示人工介入。

**重试机制**：LLM Adapter层实现了3次重试，应对网络抖动或API临时故障。

**暂停/恢复**：用户可以在Web界面暂停整个Job，修复问题后恢复执行。

不过目前的实现还比较基础，如果Agent执行一半卡住了，需要人工检查。未来可以考虑心跳检测和自动替换Agent。"

## 3. 技术选型类

### Q8: 为什么用线程而不是asyncio？

**回答要点：**
- 简单够用
- 调试友好
- 可未来迁移

**建议回答：**
"主要是权衡复杂度和收益：

**线程方案**：
- 优点：代码简单、调试容易、足够满足当前并发需求（5-10个Agent）
- 缺点：线程切换有开销，高并发下性能不如asyncio

**asyncio方案**：
- 优点：性能更好，适合大量I/O并发
- 缺点：代码复杂度高，异常处理麻烦，调试困难

我的判断是：当前场景下线程够用，而且LLM API调用本身有速率限制，不是纯性能瓶颈。所以先用线程实现MVP，验证想法正确后，如果真有性能需求再迁移到asyncio也不迟。

另外Scheduler里用ThreadPoolExecutor也保留了迁移的灵活性。"

---

### Q9: 为什么用OpenAI SDK而不是各厂商的原生SDK？

**回答要点：**
- 统一接口
- 生态兼容
- 减少依赖

**建议回答：**
"两个原因：

**统一接口**：Kimi和DeepSeek都兼容OpenAI的API格式，用OpenAI SDK可以同时支持，不需要安装多个SDK。

**代码简洁**：如果每个模型都用原生SDK，适配层代码量要翻倍。现在只需要配置不同的base_url和api_key就行。

**迁移成本低**：以后如果接入Claude、GPT-4，只要它们兼容OpenAI格式，配置一下就能用。

当然如果某个模型有独特的功能（比如MiniMax的Token Plan API），可能还是需要特殊处理。但大部分场景下OpenAI SDK够用。"

---

### Q10: Web监控为什么用轮询而不是WebSocket？

**回答要点：**
- 简单可靠
- 实时性够用
- 无状态设计

**建议回答：**
"主要是考虑到**可靠性和复杂度**的平衡：

**轮询方案**：
- 2秒轮询一次，对于监控场景完全够用
- 实现简单，不会遇到WebSocket的断线重连、心跳保活等问题
- 服务端无状态，可以随时重启

**WebSocket方案**：
- 实时性更好，但实现复杂
- 需要处理连接管理、断线重连、消息队列
- 对于MVP来说过度设计

我觉得监控数据秒级更新足够，没必要为了几百毫秒的提升增加复杂度。如果以后要做实时协作编辑那种场景，再考虑WebSocket。"

## 4. 难点挑战类

### Q11: 开发过程中遇到的最大难点是什么？

**准备1-2个真实案例，比如：**

**案例1：DAG死锁检测**
"一开始没有考虑循环依赖的问题。如果任务A依赖B，B又依赖A，调度器会死循环。

解决方案是加了就绪任务检测——如果一轮循环下来没有任务就绪，但还有未完成任务，说明有死锁，抛出异常。

```python
if not ready and len(completed) < len(tasks):
    raise Exception('Workflow deadlock detected')
```"

**案例2：Agent状态同步**
"多线程环境下Agent状态更新有竞态条件。比如Scheduler检查Agent是idle，但此时Agent刚好被分配了任务。

解决方案是用RLock保护状态更新，并且状态检查+任务分配做成原子操作。"

### Q12: 如果Agent之间的输出有冲突怎么办？

**回答要点：**
- 当前实现简单覆盖
- 可以扩展Consensus机制
- 提供人工介入

**建议回答：**
"这是目前实现还比较粗糙的地方。

**当前方案**：后完成的覆盖先完成的，冲突会记录在日志里，通过Web界面告警提示人工检查。

**设计的扩展点**：我在EventBus里留了`conflict_detected`事件，可以注册处理器实现不同的解决策略：
- **投票机制**：多个Agent投票决定
- **优先级机制**：架构师的输出覆盖开发者的
- **Mediator机制**：引入一个Senior Agent仲裁
- **人工审核**：关键决策人工确认

这部分是根据业务场景定制的，框架只提供事件通知，具体策略由业务层实现。"

---

### Q13: 如何确保任务分解的合理性？

**回答要点：**
- 模板预定义
- 人工审核
- 迭代优化

**建议回答：**
"目前有两个层面：

**Template层面**：任务分解是在`decompose_task()`方法里实现的，对于标准流程（如软件研发）是预定义的，不是动态生成的，保证基本合理。

**LLM层面**：对于需要智能分解的场景，可以调用LLM生成任务列表，然后人工审核或基于规则校验。

**监控反馈**：如果某个任务执行时间异常长，Web界面会有提示，可以人工介入调整。

未来可以考虑让LLM根据历史执行数据优化分解策略，但目前的预定义方式在软件研发场景下已经够用了。"

## 6. 代码细节类

### Q16: 讲讲AgentRegistry的实现？

**准备看代码回答：**

"AgentRegistry是Agent的注册中心，核心功能：

1. **注册/注销**：用字典存储Agent ID到实例的映射
2. **按角色查找**：维护`role_name → [agent_ids]`的反向索引
3. **状态管理**：存储每个Agent的实时状态
4. **线程安全**：用RLock保护并发访问

关键代码：
```python
def find_by_role(self, role_name: str) -> Optional[BaseAgent]:
    # 找到该角色的所有Agent，返回第一个空闲的
    agent_ids = self._by_role.get(role_name, [])
    for agent_id in agent_ids:
        agent = self._agents.get(agent_id)
        if agent and agent.get_status() == "idle":
            return agent
    return None
```

这里可以扩展负载均衡策略，比如轮询、最少任务等。"

---

### Q17: Job状态是怎么持久化的？

**回答要点：**
- JSON序列化
- 文件存储
- 恢复机制

**建议回答：**
"Job状态持久化通过Workspace组件实现：

1. **序列化**：Job和Task都实现了`to_dict()`方法，转成字典后存JSON
2. **存储**：每个Job一个JSON文件，路径`workspace/jobs/{job_id}.json`
3. **触发时机**：每次任务状态变更后自动保存
4. **恢复**：系统重启时可以`load_job()`恢复状态

代码示例：
```python
def save_job(self, job: Job):
    job_path = self.jobs_dir / f"{job.id}.json"
    job.save_to_file(str(job_path))
```

生产环境可以替换成数据库存储，但文件存储对于演示和开发足够简单可靠。"
