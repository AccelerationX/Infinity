# 研究项目联动叙事指南 —— 面试口述策略

> 本指南帮助你把分散的 15 个研究项目串联成两个有内在逻辑的故事线，让面试官看到的不只是"做过很多项目"，而是"构建了一套完整的体系"。

---

## 故事线一：AI Agent 操作系统（项目 13 → 14 → 15）

### 一句话总纲

> "我做了一个让 AI 像人一样使用电脑、调用工具，并且能从经验中不断学习进化的 Agent 操作系统。"

### 递进关系

```
13 Computer Use Agent          14 MCP Tool Hub              15 Agent Lifelong Learning
     ↓                               ↓                              ↓
"AI 怎么控制电脑？"         →   "AI 怎么调用外部工具？"    →   "AI 怎么从经验中学习并复用技能？"
"感知 + 执行 + 验证"         →   "语义匹配 + DAG 规划 + 安全" →   "模式挖掘 + 技能蒸馏 + 失败回放"
```

### 面试口述建议（3 分钟版）

"我在 Agent 方向做了三个递进的项目，最终目标是构建一个**能自主行动、能调用外部世界、并且越用越聪明的 AI Agent 操作系统**。

**第一个项目（13 号）解决的是'AI 怎么控制电脑'**。我基于 Windows UIA 做了一个桌面自动化代理，让 AI 能够观察屏幕控件树、规划鼠标键盘操作、执行点击输入，并通过像素差异和文本读取双重验证任务结果。核心难点在于 Windows 桌面环境的误触率很高，我设计了**原子化执行器**（Hover-Pause-Click）把稳定性提了上来，Demo 在 Notepad 任务上做到了 100% 成功率。

**但只会控制电脑是不够的，AI 还需要调用外部世界的能力**。于是第二个项目（14 号）我基于 Anthropic 的 MCP 协议，做了一个**生产级的多工具路由中枢**。它能够动态注册各种 MCP 服务器（文件系统、浏览器、计算器、Shell），通过 BGE 语义匹配把用户的自然语言意图映射到正确的工具，再用 DAG 规划多步骤任务，最后经过安全审计层执行。这个系统的端到端 benchmark 是 **100% 工具召回、100% 参数正确、100% 执行成功**。

**前两个项目让 Agent 能'做'，但还不能'学'**。所以第三个项目（15 号）是一个**Agent 终身学习系统**。每次 Agent 完成任务后，系统会把经验存下来，定期用 SBERT+K-Means 做模式挖掘，把相似经验蒸馏成参数化的技能模板。下次遇到同类任务时，Agent 可以直接调用已学技能，而不是每次都重新推理。我们还做了失败回放机制，把历史踩过的坑自动注入到执行上下文中，避免重蹈覆辙。最终 benchmark 是 **5.0/5.0 满分**，而且我已经写了一个 demo 把 14 号的 MCP 工具调用日志自动导入 15 号系统，实现了**从工具调用到技能沉淀的完整闭环**。

这三个项目合起来，就是一个**有手（控制电脑）、有眼耳（调用工具）、有大脑（终身学习）**的 AI Agent 的雏形。"

### 高频追问与回答

**Q：为什么这三个项目一定要分开做？不能一次性做一个大系统吗？**

**A**：每个子问题的难度和侧重点都不同。13 号的核心挑战是**工程稳定性**（桌面环境复杂、误触率高）；14 号的核心挑战是**协议设计与系统架构**（MCP 是新兴协议，需要从零搭建路由、安全、审计层）；15 号的核心挑战是**算法与学习机制**（模式挖掘、技能泛化、冲突解决）。分开做可以逐个攻克，最后再把它们像乐高一样拼起来。这也是我在 demo 中验证过的思路。

**Q：这个'操作系统'和 AutoGPT、LangChain 有什么区别？**

**A**：
- **AutoGPT** 更像一个实验性的原型，它把所有能力都耦合在一个循环里，很容易陷入无意义的自我调用循环，而且缺乏稳定的学习机制；
- **LangChain** 是一个工具编排框架，但它主要解决的是'怎么用链式 Prompt 调用工具'，没有解决'怎么在桌面上稳定执行'和'怎么从经验中提炼技能'的问题；
- 而我们的体系是**分层解耦**的：13 负责感知执行、14 负责工具路由、15 负责学习进化。每一层都可以独立运行，也可以组合使用，更像是一个模块化的操作系统架构。

**Q：如果让你把这套系统做成产品，下一步你会做什么？**

**A**：
1. **扩展感知层**：除了 UIA，引入多模态大模型做屏幕截图理解，支持更复杂的自定义 UI 和游戏界面；
2. **扩大工具生态**：把更多企业常用软件（Excel、飞书、Jira、GitHub）封装成 MCP 服务器；
3. **在线学习**：目前 15 号系统是定期批量挖掘模式，未来希望做到**流式学习**——每次成功或失败后立即更新技能库；
4. **多 Agent 协作**：把 03 号项目的多 Agent 协议引入，让多个专精 Agent 协同完成复杂任务。

## 故事线二：量化投研全栈能力（项目 06 → 07 → 08 → 09 → 10 → 11 → 12）

### 一句话总纲

> "我做了一套从底层回测引擎到上层研究工作台的完整量化投研基础设施，覆盖了策略验证、收益归因、风险诊断、市场择时、微观结构和鲁棒性评估。"

### 能力栈分层

```
┌─────────────────────────────────────────────┐
│  12 量化研究工作台                           │  ← 上层：一站式研究平台（整合全部能力）
│  （Notebook + Streamlit + 一键复现）         │
├─────────────────────────────────────────────┤
│  09 市场状态识别    10 低延迟订单簿重建        │  ← 策略层：择时 + 微观结构
│  11 对抗鲁棒性                                │  ← 策略安全与防御
├─────────────────────────────────────────────┤
│  07 交易归因分析器    08 组合风险诊断引擎      │  ← 分析层：收益从哪来 + 风险在哪
├─────────────────────────────────────────────┤
│  06 防前视偏差回测引擎                       │  ← 底层：策略验证的基础设施
└─────────────────────────────────────────────┘
```

### 面试口述建议（3 分钟版）

"我在量化方向做了七个项目，构成了一套**从底层回测到上层工作台的完整投研基础设施**。

**最底层是 06 号回测引擎**。回测是量化研究的根基，但很多研究者会犯'前视偏差'的错误——比如用收盘价做决策并在当天成交。我设计了一个严格的数据时点管理系统，每条数据都有 `available_at` 字段，策略只能使用当前时点之前的信息；同时内置了 Walkforward 验证和三档撮合仿真（Ideal / VWAP / Realistic），可以量化滑点和延迟对收益的真实影响。

**在回测之上，我需要回答两个问题：钱从哪来？风险在哪？**

**07 号交易归因分析器** 解决第一个问题。它基于 Brinson 模型把超额收益拆解为资产配置、个股选择、交互效应和交易时机；同时做了执行缺口归因，把'纸面收益'和'实盘收益'之间的鸿沟拆解为延迟成本、冲击成本、滑点和机会成本。我用它审计过实盘策略，成功定位到一个因数据字段语义误用导致 2488 bps 异常摩擦成本的 bug。

**08 号组合风险诊断引擎** 解决第二个问题。它支持多因子风险分解（MRC/ARC 严格可加性验证）、经典风险指标（VaR/CVaR/Sharpe/Sortino/MDD）、宏观情景压力测试和三级智能告警。我在审计中发现某条 `sector_rotation` 策略的 95% VaR 高达 -12.15%，为该策略的仓位控制提供了量化依据。

**再往上一层是策略研究本身。**

**09 号市场状态识别** 用隐马尔可夫模型（HMM）把市场划分为牛/熊/震荡三种 Regime，并据此动态切换最优策略配置。**10 号低延迟订单簿重建** 则从交易所原始逐笔事件中实时重建 Level-2 订单簿，提取 LOB Imbalance、VPIN 等微观结构指标。**11 号对抗鲁棒性研究** 则从反面出发，用数据投毒、特征扰动和市场操纵模拟测试策略的脆弱性，并提出集成防御方案。

**最顶层是 12 号量化研究工作台**，它把前面所有的能力整合到一个 Streamlit 界面里，支持插件化策略开发、一键复现和交互式可视化。研究员从有一个 idea 到产出完整回测报告，平均时间从 3 天缩短到 4 小时。

这七个项目不是孤立的，而是**一层一层堆叠起来的量化投研操作系统**。
"

### 高频追问与回答

**Q：你做这么多量化项目，为什么没有实盘业绩？**

**A**：这几个项目的核心目标是**构建基础设施和研究方法论**，而不是直接跑实盘赚钱。实盘业绩依赖于资金、牌照、数据、执行通道，这些资源在独立研究阶段很难获取。但我做的每个项目都在解决实盘过程中会遇到的真实问题：前视偏差怎么防、收益怎么归因、风险怎么量化、订单簿怎么重建、策略在对抗环境下是否鲁棒。这些能力是从研究到实盘的必经之路。如果有实盘数据和交易通道，这些系统可以直接对接。

**Q：如果只能挑一个项目重点讲，你推荐讲哪个？**

**A**：如果面试官是技术背景，推荐讲 **06 号回测引擎**，因为它的工程深度最高，涉及事件驱动架构、数据时点管理、Walkforward 验证等硬核内容；如果面试官是投资/研究背景，推荐讲 **07 号归因分析器** 或 **08 号风险诊断引擎**，因为它们直接回答了"收益来源"和"风险敞口"这两个投资人最关心的问题；如果面试时间很短，推荐讲 **12 号工作台**，因为它能一句话概括你的全栈能力。

**Q：你的回测引擎和 Zipline/Backtrader 有什么区别？**

**A**：Zipline 和 Backtrader 是成熟的通用回测框架，但它们有两个问题：
1. **对前视偏差的防护不够严格**：默认情况下很容易在策略中不小心用到未来信息；
2. **对 A 股市场特性的支持有限**：比如 T+1 制度、涨跌停限制、Level-2 数据格式等。

我的引擎从零设计，核心目标就是**解决这两个痛点**。它不是要替代 Zipline，而是作为一个更贴合 A 股研究需求的专用引擎。

## 跨线联动：AI 与量化的交叉点

如果你的目标岗位是 **"AI + 量化"**（比如智能投顾、另类数据挖掘、AI Trader），可以把两条故事线做一个交叉：

> "我的研究横跨两条主线。一条线是**构建 AI Agent 的感知-工具-学习能力**，让机器能够自主地完成复杂的认知和操作任务；另一条线是**构建量化投研的基础设施**，从回测、归因、风控到微观结构研究。这两条线在未来必然会交汇——**下一代量化交易员很可能就是一个 AI Agent**：它能自动读取财报、解析新闻、监控订单簿、执行交易，并从每次盈亏中不断优化自己的策略。"

这样的叙述会让面试官看到一个**有长期愿景、有体系化思考**的候选人形象。
