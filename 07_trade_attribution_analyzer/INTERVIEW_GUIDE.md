# 07 实盘交易归因分析器 —— 面试完全指南

---

## 一、项目一句话定位

这是一个**对投资组合收益进行多维度归因分析的引擎**，能够将超额收益拆解为资产配置、个股选择、行业轮动、交互项、执行摩擦等来源，并量化实施缺口（Implementation Shortfall）的各个子项。

---

## 二、核心技术栈

- **Python 3.10** + **Pandas / NumPy**
- **Brinson 归因模型**（经典投资组合归因框架）
- **Holdings-based 归因**（基于持仓的日度归因）
- **Tushare**（A 股价格与财务数据）
- **Pytest**（归因公式正确性验证）

---

## 三、核心原理

实盘策略的净值增长来自多个因素的叠加，搞清楚"钱是从哪来的、又是因为什么亏的"是量化投资的核心。本项目实现了**五维归因框架**：

### 1. Brinson 归因（5D 分解）
将组合超额收益分解为：
- **资产配置效应（Allocation）**：因为超配/低配了某些行业/资产类别而带来的收益；
- **个股选择效应（Selection）**：因为在同一行业内选对了股票而带来的收益；
- **交互效应（Interaction）**：配置和选择的协同效应；
- **交易时机效应（Timing）**：日内调仓时机带来的收益；
- **货币/其他效应（Currency / Other）**：汇率、分红再投资等。

### 2. 执行缺口归因（Implementation Shortfall Attribution）
将"纸面收益"和"实盘收益"之间的鸿沟拆解为：
- **延迟成本（Delay Cost）**：从决策到下单期间价格变动造成的损失；
- **冲击成本（Market Impact）**：订单本身对市场价格的影响；
- **滑点（Slippage）**：预期成交价和实际成交价之间的差异；
- **手续费与税费（Fees & Commissions）**；
- **机会成本（Opportunity Cost）**：因限价单未成交而错失的收益。

### 3. Holdings-based 日度归因
不依赖交易流水，仅通过每日持仓快照即可推算出归因结果，降低了数据获取门槛。

## 四、实现细节

### 架构设计
```
src/
  ├── brinson.py             # Brinson 5D 分解核心算法
  ├── execution_attribution.py # 执行缺口拆解
  ├── holdings_engine.py     # 基于持仓的日度收益计算
  ├── data_loader.py         # Tushare / 本地数据接口
  └── validator.py           # 归因可加性校验
```

### 关键模块
- `BrinsonAttributor`：输入为组合和基准在多个行业上的权重与收益率，输出 Allocation / Selection / Interaction 三项。公式严格满足 **Allocation + Selection + Interaction = 总超额收益**。
- `ExecutionAttributor`：输入为决策价格（通常是前收盘价或目标价）、预期成交量、实际成交价、实际成交量，输出延迟成本、冲击成本、滑点、机会成本四项。
- `Validator`：对每个归因结果做**可加性校验（Additivity Check）**，确保所有子项之和与总收益的差异小于 1e-6。

### 难点与解决
- **A 股 T+1 制度下的归因复杂性**：当日买入的股票不能卖出，导致持仓变化不是瞬时完成的。我们在 `holdings_engine` 中显式建模了 T+1 约束，确保买入股票在次日才能产生卖出收益。
- **数据字段语义不匹配**：在 Trading 项目的 `main_trade_execution_log.csv` 中，`system_signal` 存的是信号强度而不是决策价格。发现这个问题后，我们在数据预处理层增加了字段映射逻辑，用 `StockHistory` 的开盘价作为 `signal_price` 进行归因。

## 七、高频面试问题与回答

### Q1：Brinson 归因的三个核心项是什么？它们加起来一定等于总超额收益吗？
**A**：三个核心项是 **Allocation（配置）**、**Selection（选择）** 和 **Interaction（交互）**。在标准的 Brinson 模型中，这三项之和**严格等于**总超额收益。如果数据是日度级别的，公式在数学上是恒等式。我们项目中专门写了 `validator.py` 来做可加性校验，确保误差在 1e-6 以内。

### Q2：Implementation Shortfall 和 Slippage 有什么区别？
**A**：
- **Slippage（滑点）** 只是 Implementation Shortfall 的一个子项，指"预期成交价格"和"实际成交价格"之间的差异。
- **Implementation Shortfall（实施缺口）** 是一个更大的概念，它涵盖了从"投资决策做出"到"实际完成交易"整个过程中的所有收益损耗，包括延迟成本、冲击成本、滑点、手续费和机会成本。

### Q3：如果没有详细的交易流水，只有每日持仓，能做归因吗？
**A**：能，这就是 **Holdings-based Attribution**。核心假设是：相邻两天的持仓变化反映了交易行为。通过
```
P&L = sum(持仓_t * (价格_{t+1} - 价格_t))
```
就可以计算出日度收益，然后再用 Brinson 框架进行分解。它的精度比交易流水法稍低（无法精确到单笔交易的滑点），但在很多场景下已经足够。

### Q4：你们发现的 2488 bps 异常摩擦成本是怎么定位到的？
**A**：在对接 `main_live` 策略的交易日志时，我们发现 Execution Shortfall 高达 2488 bps，这明显不现实。通过排查数据字段，我们发现日志中的 `system_signal` 列存的是**信号强度值**（如 1.4411），而不是**目标成交价格**。我们的归因引擎把它误当作了 decision price，导致摩擦成本被极端放大。修正方法是用 `StockHistory` 中对应日期的开盘价/收盘价作为 `signal_price` 重新计算。

这个案例也体现了归因系统的一个重要价值：**它不仅能告诉我们收益从哪来，还能帮我们发现数据质量问题**。
