# 12 量化研究工作台 —— 面试完全指南

---

## 一、项目一句话定位

这是一个**面向量化研究员的一站式策略研究、回测、归因和可视化的集成工作台**，支持从数据处理到策略部署的全链路工作流。

---

## 二、核心技术栈

- **Python 3.10**（后端核心）
- **Streamlit / Gradio**（快速原型交互界面）
- **Pandas / Polars / NumPy**（数据处理）
- **Zipline / Backtrader**（策略回测，可选）
- **Plotly / ECharts**（可视化）
- **SQLite / DuckDB**（本地研究数据库）

---

## 三、核心原理

量化研究的工作流程通常非常重复：下载数据 → 清洗 → 生成因子 → 回测 → 归因 → 可视化。每个环节都需要写大量样板代码。本工作台的目标是将这些环节**模块化、自动化、可视化**，让研究员把精力集中在核心 idea 上。

### 核心设计理念
1. **Notebook + App 混合模式**：
   底层是标准的 Python Notebook/Script，保证灵活性；上层提供一个 Streamlit 应用，把常用的回测、归因、可视化操作封装成可点击的界面。

2. **插件化策略组件（Plugin-based Strategy Blocks）**：
   策略被拆分为四个标准接口：
   - `DataLoader`：数据加载
   - `SignalGenerator`：信号生成
   - `ExecutionEngine`：执行模拟
   - `RiskMonitor`：风险监控

   研究员只需要重写自己关心的模块，其余模块使用默认实现。

3. **一键复现（One-click Reproduction）**：
   每个策略研究项目都有一个 `manifest.yaml`，记录了：
   - 使用的数据版本和时间范围
   - 依赖包版本
   - 随机种子
   - 超参数配置
   这样任何时候都可以一键复现当时的回测结果。

## 四、实现细节

### 架构设计
```
workbench/
  ├── app.py                 # Streamlit 主应用
  ├── core/
  │   ├── data_loader.py
  │   ├── signal_generator.py
  │   ├── backtest_engine.py
  │   ├── attribution_engine.py
  │   └── risk_monitor.py
  ├── plugins/               # 预置策略模板
  │   ├── ma_crossover.py
  │   ├── factor_momentum.py
  │   └── pairs_trading.py
  ├── notebooks/             # 示例研究 Notebook
  └── manifest.yaml          # 一键复现配置
```

### 关键模块
- `BacktestEngine`：整合了 06 号回测引擎的核心能力，封装成更简单的接口。研究员只需传入 `signal_df`（包含 date、symbol、signal 列），引擎自动完成撮合和收益计算。
- `AttributionEngine`：整合了 07 号归因分析器的 Brinson 和执行缺口归因，输出可交互的 HTML 报告。
- `PluginManager`：动态扫描 `plugins/` 目录，自动注册策略模板。支持热更新，新增策略无需重启应用。

### 难点与解决
- **UI 和底层代码的耦合**：初期想把所有逻辑都写到 Streamlit 里，导致代码难以测试。解决方式是**严格分层**：底层 `core/` 是纯 Python 模块，完全可脱离 UI 运行；`app.py` 只做界面绑定和参数传递。
- **数据版本管理**：研究过程中数据经常更新（比如财报修正、除权除息调整）。我们通过在 `manifest.yaml` 中记录数据的 MD5 校验码，确保复现时使用和当时完全一致的数据版本。

## 七、高频面试问题与回答

### Q1：这个工作台和市面上的量化平台（如 JoinQuant、RiceQuant）有什么区别？
**A**：主要区别在三个层面：
1. **定位不同**：JoinQuant 是面向散户的云端平台，而我们这个工作台是面向**专业研究员的本地研究工具**，更注重灵活性和可扩展性；
2. **数据可控**：本地运行意味着可以用自己的私有数据源，不用担心数据安全和 API 限制；
3. **模块化设计**：市面上的平台策略结构通常是固定的，而我们的工作台把策略拆成了四个可插拔模块，研究员可以只重写自己关心的部分。

### Q2：一键复现是怎么实现的？如果数据更新了怎么办？
**A**：每个策略研究项目都会生成一个 `manifest.yaml`，里面记录了：
- 数据文件的路径和 MD5 校验码
- Python 依赖包的精确版本
- 随机种子和超参数
- 代码的 Git commit hash

复现时，工作台会先检查数据 MD5 是否一致。如果数据已经被更新，会提示用户选择"使用最新数据重新运行"或"从历史备份中恢复原始数据版本"。

### Q3：你们怎么保证回测结果的可信度？
**A**：可信度来自两个层面：
1. **引擎层面**：底层回测引擎是 06 号项目，具备严格的 `available_at` 时点管理和 Walkforward 验证，防止前视偏差；
2. **流程层面**：工作台强制要求策略研究项目包含"敏感性分析"和"压力测试"两个 Notebook，避免研究员只看一条最优净值曲线就做决策。

### Q4：这个项目最大的工程挑战是什么？
**A**：最大的挑战是**在灵活性和易用性之间找平衡**。如果封装得太死，高级用户会觉得束手束脚；如果放得太开，新手又不知道怎么用。我们的解决方案是**分层设计**：底层 `core/` 是完全开放的 Python API，高级用户可以直接调用；上层 `app.py` 只暴露最常用的 20% 功能，覆盖 80% 的使用场景。
