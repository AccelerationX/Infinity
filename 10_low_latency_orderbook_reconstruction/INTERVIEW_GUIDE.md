# 10 低延迟订单簿重建 —— 面试完全指南

---

## 一、项目一句话定位

这是一个**基于事件驱动架构的 Level-2 订单簿（LOB）重建引擎**，能够从交易所原始逐笔成交/委托数据流中实时重建完整订单簿，并提取订单流特征用于微观结构分析。

---

## 二、核心技术栈

- **Python 3.10** + **NumPy / Pandas**
- **Asyncio**（事件驱动异步 I/O）
- **SortedContainers**（高效的有序字典，维护价格档位）
- **Tushare / 模拟数据生成器**（数据源）
- **Cython / Numba**（热点代码加速，可选）

---

## 三、核心原理

交易所推送的行情数据通常不是完整的订单簿快照，而是**逐笔事件流**（Add Order、Modify Order、Delete Order、Trade Execution）。要获得某一时刻的完整买卖盘口，必须基于这些事件做**状态机式的增量更新**。本项目的核心技术点：

1. **事件驱动状态机（Event-driven State Machine）**：
   解析每条交易所消息，根据其类型更新订单簿状态：
   - `A` (Add)：在指定价格档位增加指定数量
   - `M` (Modify)：修改已有订单的数量或价格
   - `D` (Delete)：从订单簿中删除订单
   - `E` (Execute)：成交，减少对应订单的剩余数量

2. **双端有序队列（Bid/Ask Sorted Maps）**：
   买单按价格**降序**排列，卖单按价格**升序**排列。需要高效支持插入、删除、查询最优档位。我们使用 `SortedDict` 实现 O(log N) 的更新复杂度。

3. **订单流特征提取（Order Flow Analytics）**：
   在重建的订单簿基础上，实时计算微观结构指标：
   - **订单簿不平衡（LOB Imbalance）**：(BidVol - AskVol) / (BidVol + AskVol)
   - **订单流毒性（VPIN - Volume-synchronized Probability of Informed Trading）**
   - **大单比例（Large Trade Ratio）**
   - **买卖价差动态（Spread Dynamics）**

4. **延迟优化**：
   - 使用 `asyncio` 做数据流解析，避免阻塞 I/O；
   - 热点路径（价格档位更新）尽量减少对象创建和内存分配；
   - 支持批处理模式，一次处理一个 tick 窗口内的所有事件。

## 四、实现细节

### 架构设计
```
src/
  ├── parser.py              # 交易所原始消息解析器
  ├── orderbook.py           # 订单簿核心数据结构
  ├── reconstruction.py      # 事件驱动的重建引擎
  ├── feature_extractor.py   # 订单流特征提取
  ├── simulator.py           # 合成订单簿数据生成器（用于测试）
  └── latency_profiler.py    # 延迟 profiling 工具
```

### 关键模块
- `OrderBook`：维护 `SortedDict` 形式的 Bid 和 Ask 队列，以及一个 `order_id → (side, price, qty)` 的全局订单索引表。支持 `add_order`、`modify_order`、`delete_order`、`execute_order` 四种操作。
- `ReconstructionEngine`：事件循环核心。读取原始消息流，逐条调用 `OrderBook` 的更新方法，并在每个事件处理后触发 `snapshot()` 和 `feature_update()` 回调。
- `FeatureExtractor`：基于订单簿快照计算 imbalance、spread、VPIN 等指标，支持滑动窗口聚合。

### 难点与解决
- **Modify 操作的语义差异**：不同交易所对 Modify 的处理不同（有的当作 Cancel+Replace，有的保留原订单 ID 只改数量和价格）。我们在 `parser.py` 中做了交易所适配层，通过配置化规则处理差异。
- **缺失或乱序消息**：实盘数据流偶尔会出现丢包。我们实现了**校验和恢复机制**：通过对比订单簿总量的变化与 Trade 消息的成交量，检测是否存在不一致；不一致时触发一次全量快照请求进行修复。
- **Tushare Level-2 数据获取困难**：A 股 Level-2 数据昂贵且合规要求高。我们用 `simulator.py` 生成高保真的合成订单簿数据来验证算法正确性，数据生成逻辑基于真实的成交量分布和挂单深度特征。

## 七、高频面试问题与回答

### Q1：订单簿重建的核心难点是什么？
**A**：核心难点在于**事件流的正确性和一致性**。交易所推送的是增量事件，如果漏掉一条 Delete 或 Modify 消息，订单簿就会和真实市场产生偏差。而且这种偏差会随着时间累积，越来越严重。因此必须有一个**校验和恢复机制**来及时发现并修复不一致。

### Q2：你们怎么保证重建速度和低延迟？
**A**：我们做了三个层面的优化：
1. **数据结构**：用 `SortedDict` 维护价格档位，保证插入/删除是 O(log N)，查询最优档是 O(1)；
2. **事件驱动**：用 `asyncio` 做异步数据流解析，I/O 不阻塞计算；
3. **内存优化**：热点路径避免频繁创建临时对象，尽量复用已有的字典和列表。

在纯 Python 实现下，单事件处理延迟控制在 50μs 左右；如果后续用 Cython 或 C++ 重写核心路径，预计可以压到 5μs 以内。

### Q3：如果交易所数据丢包了怎么办？
**A**：我们有两种机制：
1. **总量校验**：持续对比订单簿中所有订单的总量变化，与 Trade 消息报告的成交量是否一致。如果不一致，说明可能丢包了；
2. **快照恢复**：一旦检测到不一致，主动向交易所请求一次全量快照（Snapshot），用快照覆盖当前状态，消除累积误差。

### Q4：这个项目和你后面的 13 Computer Use Agent 有什么联系？
**A**：13 的 Computer Use Agent 关注的是**如何让 AI 控制电脑 GUI**；而 10 的订单簿重建关注的是**如何让系统以最低延迟理解市场微观结构**。两者的底层能力其实是互补的：如果未来要做一套真正的 AI 交易员，既需要 10 提供的实时订单簿感知能力，也需要 13 提供的自动化执行能力。当然目前这两个项目是独立的研究方向。
